# تصميم: محفظة المصنع (Factory Wallet) + أجور العمال

**التاريخ:** 2026-07-18
**الحالة:** مسودة بانتظار مراجعة العميل

---

## 1. الخلفية والدافع

المعرض الرئيسي "مازايا" هو **صاحب المصنع**. الحاج بيحوّل للمصنع فلوس يوميًا
("تحويل للمصنع"). الراجل اللي في المصنع أمين، بس صاحب المعرض مش تكنولوجي وعايز
يشوف بوضوح:

- بعتلنا فلوس قَدّ إيه؟
- اتصرَف قَدّ إيه وفين؟
- الرصيد الحالي قدّ إيه؟ (ممكن بالسالب لو صرف أكتر من اللي اتبعت)

صفحة `/journal` الموجودة فيها الحركة كلها، بس مختلطة مع كروت المخزون والأوردرات
والتقارير اليومية/الأسبوعية، فمش واضحة كـ"محفظة". محتاجين صفحة نقية بتعرض
المحفظة منفصلة، مع جدول يومي تراكمي يفهمه الحاج بسرعة.

كمان محتاجين صفحة منفصلة لأجور العمال (سلف/قبض) بدل ما تكون مدفونة داخل
النثريات.

## 2. الأهداف

1. **محفظة واضحة للمصنع**: رصيد تراكمي صحيح = وارد − مصروف − مدفوعات.
2. **عرض يومي**: كروت + جدول، لكل يوم (بداية اليوم / وارد / مصروف / رصيد النهاية).
3. **تفاصيل عند الطلب**: الضغط على أي يوم يفتح حركاته (نوع/بيان/جهة/مبلغ).
4. **فصل التمريري**: تحويل المعرض للمورد مباشرةً يدخل في `/journal` و`/budget`
   بس، **مش في المحفظة** — لأن الفلوس معدّتش على المحفظة أصلًا.
5. **أجور العمال منفصلة**: نوع قيد جديد `أجور عمال` يدخل في المصروف تلقائيًا،
   وصفحة `/workers` تفرّق بين سلفة/قبض.

## 3. النطاق (Out of scope)

- **لا رصيد افتتاحي خاص**: العميل يسجّل أول حركة كـ "دفعة واردة من معرض" عادية.
- **لا ترحيل للقيود القديمة**: تجاهل مؤقت لحد ما يتحدد وضع العميل. (المحفظة
  تحسب تراكميًا من أول حركة موجودة، بس نوضّح للعميل إن أي قيود قديمة هتأثر.)
- **لا تغيير في `/journal`**: الإدخال يبقى من هناك كما هو.
- **لا تكرار بيانات**: المحفظة = قراءة لـ `journal_entries`، بدون جداول جديدة.

## 4. العمارة

### 4.1 مبدأ أساسي: المحفظة = قراءة لـ `journal_entries`

كل العمليات في النظام بتسجّل قيدًا في `journal_entries` تلقائيًا:

| العملية | entry_type | المصدر |
|---|---|---|
| شراء ألواح/إكسسوارات | `مشتريات` | `/api/boards/purchase`, `/api/accessories/purchase` |
| نثريات | `نثريات` | `/api/overhead` |
| **أجور عمال** (جديد) | `أجور عمال` | `/api/overhead` (شوف 4.4) |
| دفعة من معرض | `دفعة واردة من معرض` | `/api/journal` |
| دفع لمورد مباشر | `دفعة صادرة لمورد` | `/api/journal` |
| تحويل تمريري | قيدين `is_pass_through=true` | `/api/journal` |

المحفظة بتجمع القيود وتحسب الرصيد. **مفيش جدول جديد ولا تكرار.**

### 4.2 معادلة الرصيد (تعديل `src/lib/finance.ts`)

```
الرصيد = (وارد حقيقي) − (مصروف حقيقي) − (مدفوعات حقيقية)

وارد حقيقي      = entry_type='دفعة واردة من معرض' AND is_pass_through=false
مصروف حقيقي     = entry_type IN ('مشتريات','نثريات','أجور عمال') AND is_pass_through=false
مدفوعات حقيقية  = entry_type='دفعة صادرة لمورد' AND is_pass_through=false
```

التمريري (أي قيد `is_pass_through=true`) **مستثنى تمامًا** من حسابات المحفظة.
الفرق الوحيد بين `/journal` و`/budget` (اللي بيشملوا التمريري) والمحفظة هو
الفلتر ده.

### 4.3 صفحة `/factory-wallet` (جديد)

صفحة Client Component، **قراءة فقط**. كل الإدخال من `/journal`.

#### رأس الصفحة: عنوان + الفلتر
- العنوان: "محفظة المصنع".
- الفلتر (اختياري): زرار "آخر 7 أيام" (افتراضي) / "آخر 30 يوم" / فلتر مخصص
  (من/إلى). البحث بالوصف اختياري.

#### الكروت الأربعة الثابتة (فوق الجدول) — **اليوم الحالي فقط**

أربعة كروت ثابتة دائمًا في أعلى الصفحة، بتحمل أرقام **اليوم الحالي فقط**
(مش لكل يوم، مش فيها تكرار):

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│  بداية اليوم  │    وارد      │    مصروف      │ رصيد النهاية  │
│              │   (اليوم)    │   (اليوم)     │   (اليوم)    │
│   -1,500     │   +2,000     │    -800      │    -300      │
│              │  🟢 2 حركة   │  🔴 3 حركة   │              │
└──────────────┴──────────────┴──────────────┴──────────────┘
        📅 الأربعاء 17 يوليو 2026 (تاريخ اليوم الحالي)
```

- **بداية اليوم** = `calcOpeningBalance(allRows, todayKey)` — رصيد آخر يوم قبله
  (تراكمي من أول النظام لحد نهاية أمس). دي نقطة الانطلاق لليوم.
- **وارد (اليوم)** = `calcIncome(todayRows)` — مجموع `entry_type='دفعة واردة من معرض'`
  لليوم الحالي فقط.
- **مصروف (اليوم)** = `calcExpense(todayRows) + calcPayout(todayRows)` — مجموع
  `مشتريات + نثريات + أجور عمال + دفعة صادرة لمورد` لليوم الحالي فقط.
  المدفوعات للموردين تدخل هنا لأنها فلوس خرجت من المحفظة. لو فيه مدفوعات
  للموردين في اليوم، نبيّن تحت رقم المصروف سطر صغير: "منها X للموردين"
  (معلومة فقط، الحساب شاملها بالفعل).
- **رصيد النهاية (اليوم)** = `calcClosingBalance(allRows, todayKey)` = بداية +
  وارد − مصروف. ده هو **الرصيد الحالي للمصنع**.

> الكروت دي **لحظية** — بتتحدّث أول ما تفتح الصفحة (وقت كل refresh) علشان
> تعكس وضع اليوم الحالي. مفيهاش إمكانية الضغط/التوسيع.

#### الجدول التفصيلي (تحت) — كل الأيام في النطاق
جدول بصف لكل يوم فيه حركات (داخل نافذة الفلتر — افتراضيًا آخر 7 أيام)، مرتب
تنازليًا (الأحدث فوق):

| التاريخ | بداية اليوم | وارد | مصروف | رصيد النهاية |
|---|---|---|---|---|
| الأربعاء 17 يوليو | -1,500 | +2,000 | -800 | **-300** |
| الثلاثاء 16 يوليو | ... | ... | ... | ... |

- **الضغط على أي صف** يوسّع تفاصيل الحركات لليوم ده (نوع/بيان/جهة/طريقة دفع/
  مبلغ) — نفس المنطق الموجود في `_journal-page-wrapper.tsx` لـ"تقرير اليوم".
- العمود الأخير (رصيد النهاية) ملوّن: أخضر موجب، أحمر سالب.

> ملاحظة: الكروت الأربعة الثابتة بتعكس **اليوم الحالي**، أما الجدول فيعرض
> **كل الأيام** (بما فيها اليوم الحالي كصف عادي). لو حاب يركّز على يوم غير
> اليوم الحالي، يضغط على صفه في الجدول.

#### زر التصدير
`exportToExcel` للجدول اليومي (موجود أصلاً في `src/lib/excel.ts`).

### 4.4 أجور العمال (نوع قيد جديد + تطوير `/workers`)

#### 4.4.1 تعديل `src/lib/finance.ts`
```ts
export const EXPENSE_TYPES = ['مشتريات', 'نثريات', 'أجور عمال'] as const;
```
+ إضافته في `VALID_ENTRY_TYPES` (يتم تلقائيًا لأنه بياخد من `EXPENSE_TYPES`).

#### 4.4.2 تعديل `src/lib/format.ts`
```ts
ENTRY_TYPE_LABELS['أجور عمال'] = 'أجور عمال';
ENTRY_TYPE_COLORS['أجور عمال'] = 'bg-amber-100 text-amber-700 border-amber-300';
```

#### 4.4.3 تعديل `/api/overhead`
السلوك الحالي: لما `worker_id` موجود، القيد بيتسجل بـ `entry_type='نثريات'`.
السلوك الجديد:
- لو `worker_id` موجود + `category === 'أجور عمال'` → القيد بـ `entry_type='أجور عمال'`.
- غير كده → `entry_type='نثريات'` (زي ما هو).
- نضيف حقل `payment_kind` (اختياري) في `overhead_expenses`:
  `'قبض' | 'سلفة'` (افتراضي `'قبض'`). ده علشان نقدر نفرّق في صفحة
  العمال بين الدفعة النهائية والسلفة. (الكل بيدخل في المصروف على المحفظة.)

#### 4.4.4 تعديل صفحة `/workers` — إضافة tab "أجور العمال"
صفحة `/workers` الحالية تبقى بتبوين (tabs):
- **Tab 1: "العمال"** (الموجود حاليًا): قائمة العمال + الإحصائيات.
- **Tab 2: "أجور العمال"** (جديد): إدارة أجور كل عامل.

محتوى tab "أجور العمال":
- لكل عامل بطاقة/صف: الاسم + مجموع القبض + مجموع السلف + الرصيد الصافي
  (قبض − سلف) + تاريخ آخر دفعة.
- **الضغط على العامل** يفتح سجل تفصيلي (التاريخ/النوع [قبض أو سلفة]/المبلغ/ملاحظة)
  في نفس الصفحة (expandable row أو drawer).
- **زر "دفعة لعامل"** يفتح فورم صغير: العامل/المبلغ/نوع الدفع (قبض/سلفة)/تاريخ/
  ملاحظة → POST `/api/overhead?create_journal=true` مع `worker_id` +
  `category='أجور عمال'` + `payment_kind`.
- زر تصدير Excel للسجل.

#### 4.4.5 Migration
- إضافة عمود `payment_kind` إلى جدول `overhead_expenses` (nullable، default null).
- الـ migration بـ Prisma.

## 6. خطة الـ API

### 6.1 جديد: `GET /api/factory-wallet`
- **Params:** `days` (default 7)، أو `date_from`/`date_to`.
- **Response:**
```json
{
  "ok": true,
  "data": {
    "today": {
      "date": "2026-07-18",
      "opening": -1500,
      "income": 2000,
      "expense": 800,
      "payout": 0,
      "closing": -300,
      "count": 5
    },
    "days": [
      {
        "date": "2026-07-17",
        "opening": -1500,
        "income": 2000,
        "expense": 800,
        "payout": 0,
        "closing": -300,
        "count": 5,
        "entries": [ /* القيود الكاملة ليوم 17 — للجدول الموسع */ ]
      },
      ...
    ]
  }
}
```
- **`today`**: ملخّص **اليوم الحالي** للكروت الأربعة الثابتة (opening/income/
  expense/closing). لو مفيش حركات اليوم، الأرقام = 0 وclosing = opening.
- **`days`**: الأيام في نافذة الفلتر للجدول التفصيلي، مرتبة تنازليًا.
- **المنطق:**
  1. نجيب كل القيود (`is_pass_through=false`) مرتبة `date ASC`.
  2. نحسب رصيد كل يوم تراكميًا (نفس منطق `calcOpeningBalance` + `calcClosingBalance`).
  3. للـ pagination/الأداء: نحسب الرصيد الإجمالي لكل الأيام قبل نافذة العرض
     مرة واحدة (running balance من بداية النظام)، وبعدين نعرض الأيام داخل
     النافذة.
  4. `today` = ملخص اليوم الحالي (يكون أول صف في الجدول كمان لو داخل النافذة).
- **SQL:** بنستخدم `$queryRawUnsafe` مع params parameterized (نفس نمط `/api/journal`).

## 9. الاختبارات

### 9.1 وحدات (unit)
- `finance.ts`: التمريري مستثنى، الأنواع الجديدة محسوبة.
- معادلة الرصيد: مثال رقمي (بداية −500، وارد 200، تمريري 1000، مصروف 300 →
  الرصيد = −500+200−300 = −600، التمريري ملوش تأثير).

### 9.2 تكامل (integration)
- `GET /api/factory-wallet` برجع الأيام الصحيحة بترتيبها وأرصدتها.
- POST على `/api/overhead` بـ `worker_id` + `category='أجور عمال'` بينشأ قيد
  بـ `entry_type='أجور عمال'`.

### 9.3 يدوي
- سيناريو كامل: سجّل دفعة واردة 1000 → سجّل شراء 400 → سجّل تمريري 500
  للمورد → افتح المحفظة: الرصيد = 600 (1000−400)، التمريري مأثرش.

## 11. قرارات معلّقة / أسئلة مفتوحة

- **القيود القديمة**: العميل قال "تجاهلها لحد ما نشوف وضعه". التنفيذ الحالي
  بيحسب تراكمي من أول قيد. لو العميل لاحقًا قرّر يبدأ من تاريخ، نضيف فلتر
